## 1. プロジェクト概要

### 目的
12期生が作成した家計簿アプリ（gen12）を土台に、11期各チームが **Step3-2 で培ったプロダクトのエッセンス**（くんよみ・MOF・wasabimonja 等の知見や機能アイデア）を反映し、**家計簿アプリとしての質**を一段高める。その結果物を 12期生に示し、「半年後の姿」として技術と開発のレベル感を伝える。

### 技術スタック
- **フロントエンド**: Next.js + TypeScript + Tailwind CSS
- **バックエンド**: FastAPI + Python
- **データベース**: Azure MySQL
- **デプロイ**: Azure
- **AI機能**: OpenAI API（継続利用）

## 2. 機能要件

機能のサマリは別紙 [機能一覧](./機能一覧.md) を参照。

### 2.1. 必須機能（Sprint 1で完了目標）

gen12の機能を11期レベルで再実装する。認証により、支出・分析をログインユーザーに紐づける。

#### 2.1.1. 認証機能
- **概要**: Google認証によるログイン・サインアップ・セッション管理。支出記録・分析を「ログイン中のユーザー」に紐づける。
- **エンドポイント**:
  - `POST /api/auth/google`（Google認証によるログイン/サインアップ）
  - `POST /api/auth/logout`（ログアウト）
  - `GET /api/auth/me`（現在ユーザー取得）
- **機能詳細**:
  - **Google OAuthによる認証のみ**（メール/パスワードによる認証は実装しない）
  - ログイン/新規登録をGoogle認証に一本化
  - サーバー側 セッションまたはJWTによる認証状態の保持（14日）
  - 未認証時はログイン画面へリダイレクト（またはログイン促し）
  - **認証情報からユーザーを取得**（URLパラメータでのuser指定は使用しない）
- **gen12との違い**: gen12はユーザー名の入力のみ。本アプリでは認証により「誰のデータか」を一意にし、他ユーザーのデータを参照・改ざんできないようにする。
- **コメント**
  - スマホアプリだとJWTの方が一般的だそうです。 -- maccha

#### 2.1.2. 支出記録機能
- **概要**: アイテム名、金額を入力して記録
- **エンドポイント**: `POST /api/expense`
- **機能詳細**:
  - **認証情報からユーザーを取得**（URLパラメータでのuser指定は使用しない）
  - アイテム名、金額の入力
  - 日付の自動記録（デフォルトで現在日時を挿入、変更可能）
  - カテゴリ選択（デフォルトは未設定）
  - データベースへの保存
  - 保存時にスナックバーで「登録完了」表示
  - 単発保存とまとめ保存の両対応
- **入力フロー**:
  - **手入力**: ホーム画面→モーダル表示→日付・商品名・金額・カテゴリ入力→保存
  - **レシート読み取り**（Sprint 2以降）: 撮影→複数品目の同時検出→検出結果を追加→一括保存可能、誤操作時はモーダルを閉じて戻れる
- **gen12との違い**:
  - CSV → データベース（Azure MySQL）に変更
  - 型安全性の確保（TypeScript + Pydantic）
  - エラーハンドリングの強化
- **コメント**
  - 誤投稿時に修正できる機能追加も見据えたコード設計ができると尚良いと思います。--maccha

#### 2.1.3. 分析機能（AIコーチング）
- **概要**: ログインユーザーの合計金額を計算し、AIコーチングを提供
- **エンドポイント**: `GET /api/analyze`
- **機能詳細**:
  - **認証情報からユーザーを取得**（URLパラメータでのuser指定は使用しない）
  - ログインユーザーの合計金額を計算
  - AIコーチング機能（鬼コーチ/天使コーチ）
  - **コーチモードは設定で事前選択可能**（分析時の都度選択は不要）
  - 目標支出額との比較（予算設定時）
  - エンプティ状態の扱い（データなし時の表示）
  - 再分析パスの制御
- **コーチタイプ**:
  - `oni`: 鬼コーチ（厳しい指導）
  - `tenshi`: 天使コーチ（優しい指導）
- **gen12との違い**:
  - データベースからの集計（CSV → DB）
  - 予算設定の柔軟化（ハードコード → DB保存）
  - 認証情報からユーザー取得によりセキュリティ強化
- **コメント**
  - 自動で定期的に、メールかLINEなど送信（今後実装）
  - サーバー側で認証するので、フロントからユーザー名を指定する必要があるのか確認 --maccha  

#### 2.1.4. 基本UI
- **概要**: 支出記録と分析結果を表示する画面（Sprint 1ではホーム画面をMVP対象とする）
- **画面設計**: Figma/Figma Makeベースのデザインを正として実装
- **ホーム画面**（Sprint 1で実装）:
  - 日付に連動した進捗バー（日付ベースでの進捗管理）
  - 月次予算の「設定額」「消費額」「残額」を表示
  - 予算状態に応じた色変化（赤＝オーバー、オレンジ＝残り20%以下など）
  - コーチ（天使/鬼）による軽い挨拶メッセージ（ジャブ的コーチング）
  - 直近レポート（週次レポート）：月曜日にメール等で配信される要約を表示
  - メイン機能ボタンはモーダル表示想定（手入力、レシート読み込み、コーチング、予算設定）
  - ホームに内訳を置く案の検討（コーチ重視の体験とのバランスで一旦試験的に入れる）
- **ダッシュボード**（Sprint 2以降）:
  - 月次予算の集計表示。カテゴリ色分け（食費、日用品、趣味娯楽、交通費、水道光熱費、二分類）を採用
  - 目標達成/未設定/予算オーバー等のステータス表示
  - 分析機能の入口（データなしならエンプティ表示、再分析も可能）
  - 予算未設定時でもダッシュボードから設定可能
  - カテゴリ別の内訳詳細（矢印押下で展開）
  - 絞り込み機能（検索、カテゴリ別、日付別）。モーダル上にモーダルを重ねるUI想定
- **設定画面**（Sprint 1ではコーチ切替は後続で対応）:
  - ニックネーム編集
  - コーチモード（天使/鬼）選択は初期設定で事前選択可能。分析の都度選択する手間は廃止
- **gen12との違い**:
  - モダンなUI/UX（Next.js + Tailwind CSS）
  - レスポンシブ対応
  - 型安全なコンポーネント（TypeScript）
  - Figma Makeのソースコードをベースに実装


#### 2.1.5. ユーザー登録／ニックネーム設定機能
- **概要**: Google認証でユーザー登録、ニックネームを設定
- **機能詳細**:
  - Google認証による新規登録（ログイン/サインアップを一本化）

- **画面構成**: ニックネーム、各種設定の選択画面

## 4. API設計

### 4.1. 認証API
- **エンドポイント**: `POST /api/auth/google`
- **機能**: Google認証によるログイン/サインアップの一本化
- **リクエストボディ**:
  ```json
  {
    "id_token": "Google IDトークン"
  }
  ```
- **レスポンス**:
  ```json
  {
    "user": {
      "id": 1,
      "email": "user@example.com",
      "nickname": "しゅんすけ"
    },
    "token": "JWTトークンまたはセッションID"
  }
  ```
- **エンドポイント**: `POST /api/auth/logout`
- **機能**: ログアウト（セッション/JWTの無効化）
- **エンドポイント**: `GET /api/auth/me`
- **機能**: 現在のログインユーザー情報を取得
- **認証**: 認証情報（セッション/JWT）からユーザーを取得
- **レスポンス**:
  ```json
  {
    "id": 1,
    "email": "user@example.com",
    "nickname": "しゅんすけ",
    "coach_mode": "tenshi"
  }
  ```
- **エラーハンドリング**:
  - 400: バリデーションエラー（トークン不正等）
  - 401: 未認証（`/api/auth/me`の場合）
  - 500: サーバーエラー

### 4.2. 支出記録API
- **エンドポイント**: `POST /api/expenses`
- **認証**: 認証情報（セッション/JWT）からユーザーを取得（URLパラメータでのuser指定は使用しない）
- **リクエストボディ**:
  ```json
  {
    "item": "コーヒー",
    "price": 500,
    "expense_date": "2026-01-28",
    "category_id": null
  }
  ```
- **レスポンス**:
  ```json
  {
    "id": 1,
    "user_id": 1,
    "item": "コーヒー",
    "price": 500,
    "expense_date": "2026-01-28",
    "category_id": null,
    "created_at": "2026-01-28T10:00:00Z"
  }
  ```
- **エラーハンドリング**:
  - 400: バリデーションエラー（必須項目不足、型不一致等）
  - 401: 未認証
  - 500: サーバーエラー

### 4.3. 分析API
- **エンドポイント**: `GET /api/analyze`
- **認証**: 認証情報（セッション/JWT）からユーザーを取得（URLパラメータでのuser指定は使用しない）
- **機能詳細**:
  - ログインユーザーの合計金額を計算
  - コーチモードは設定から取得（分析時の都度選択は不要）
  - AIコーチング機能（OpenAI API）
  - 予算との比較（予算設定時）
- **レスポンス**:
  ```json
  {
    "user": "しゅんすけ",
    "total": 15000,
    "budget": 20000,
    "coach_message": "よく頑張っています！...",
    "coach_type": "tenshi"
  }
  ```
- **エラーハンドリング**:
  - 401: 未認証
  - 404: ユーザーが見つからない
  - 500: サーバーエラー

### 4.4. 履歴取得API（追加機能）
- **エンドポイント**: `GET /api/expenses`
- **認証**: 認証情報（セッション/JWT）からユーザーを取得（URLパラメータでのuser指定は使用しない）
- **クエリパラメータ**:
  - `page`: ページ番号（オプション、デフォルト: 1）
  - `per_page`: 1ページあたりの件数（オプション、デフォルト: 20）
- **レスポンス**:
  ```json
  {
    "expenses": [
      {
        "id": 1,
        "item": "コーヒー",
        "price": 500,
        "expense_date": "2026-01-28"
      }
    ],
    "total": 100,
    "page": 1,
    "per_page": 20
  }
  ```

- **エンドポイント**: `GET /api/budgets`
- **認証**: 認証情報（セッション/JWT）からユーザーを取得
- **レスポンス**: 上記と同じ形式

### 4.6. レシート画像解析API（11期機能統合）
- **エンドポイント**: `POST /api/receipts/analyze`
- **リクエスト**:
  - Content-Type: `multipart/form-data`
  - `file`: 画像ファイル（PNG, JPG, JPEG）
- **レスポンス**:
  ```json

    "price": 500,
    "date": "2026-01-28",

  }
  ```
- **エラーハンドリング**:
  - 400: ファイル形式不正、ファイルサイズ超過
  - 500: サーバーエラー、AI解析エラー

### 4.8. カテゴリAPI（11期機能統合）
- **エンドポイント**: `GET /api/categories`
- **レスポンス**:
  ```json

    {
      "id": 1,

      "level": 1

  ]
  ```

### 4.9. キーワード検索API（11期機能統合）
- **エンドポイント**: `GET /api/expenses/search?keyword={keyword}`
- **クエリパラメータ**:
  - `keyword`: 検索キーワード（必須）
- **レスポンス**:
  ```json

    "results": [
  ```

準備期。**機能一覧・データ定義を先に固め、テーブルは最初に作成する**。途中からのマイグレーションはチーム全体の整合を取りづらいため、Sprint 0 でスキーマを確定し、Sprint 1 では「テーブルを触らない前提」で開発を始められるようにする。

- [ ] **機能一覧の作成・確定**
  - 本ドキュメント「2. 機能要件」をベースに、機能一覧を作成
  - Sprint 1 で実装する機能と Sprint 2 で実装する機能の切り分けを明記
- [ ] **データ定義・テーブル定義の確定**
  - 本ドキュメント「3. データモデル」をベースに、全テーブル定義を確定
  - Sprint 1 で使うテーブルと Sprint 2 で追加するテーブルの切り分けを明記
- [ ] **テーブルの作成（マイグレーション実行）**
  - 確定した定義で Azure MySQL 上にテーブルを作成

  - ※途中からのマイグレーションを避けるため、必要なテーブルは Sprint 0 で作成しておく
- [ ] **基本要件定義書の最終確認**
  - 本ドキュメント全体のレビュー
  - 機能要件・データモデル・API設計の整合性確認

### Sprint 1（2/4 〜 2/10）
Sprint 0 でテーブル・モックが揃っている前提で、必須機能の実装に集中する。**ホーム画面をMVP対象とし、ダッシュボードはSprint 2以降で実装する。**

## 9. 制約事項

### 9.1. 技術的制約
- Azure MySQLを使用（データベース選択の制約）
- OpenAI APIの継続利用（gen12と同様）

### 9.2. スケジュール制約
- 最終納期: 2月17日
- チームメンバーの疲弊状態を考慮した省エネ開発

### 9.3. リソース制約
- 最小限のリソースで最大の成果を出す
- AI駆動ワークフローによる効率化

### 9.4. 開発方針（データベース）
- **テーブルは Sprint 0 で定義・作成する**
- 前回開発では途中からのマイグレーションが負荷となり、チーム全体でスキーマを揃えるのが難しかった経験がある
- そのため、必要なテーブル定義を Sprint 0 で確定し、Sprint 1 開始時点でテーブルが存在している状態にする
- Sprint 1 以降は原則として新規テーブル追加のみ（既存テーブルの変更は最小限に抑える）
